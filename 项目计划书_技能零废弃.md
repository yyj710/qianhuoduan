## 一、项目概述

### 1.1 项目背景

在"斜杠青年"、"副业文化"、"零工经济"快速发展的背景下，越来越多的大学生、自由职业者乃至退休人员，拥有多种非主业的碎片化技能——PPT 美化、数据整理、方言翻译、短视频剪辑、简历优化、手绘插画、小型编程脚本、留学文书润色等。然而，这些技能往往因**缺乏展示渠道、信任机制缺失、交易平台门槛过高**而被闲置，形成严重的"技能浪费"。

与此同时，校园内存在大量**轻量、灵活、低成本、快速响应**的服务需求，但传统自由职业平台（猪八戒、Upwork 等）存在佣金高（15%-20%）、流程重、响应慢、缺乏校园信任等问题，难以覆盖碎片化、小额、非标准化的"微任务"。

**核心痛点**：

|痛点|供给方（技能者）|需求方（求助者）|
|-|-|-|
|渠道缺失|有技能无处展示|有需求找不到人|
|信任薄弱|担心被坑、收不到钱|担心质量差、被骗|
|成本过高|校外平台佣金 15%-20%|小额需求无人接|
|匹配低效|曝光靠运气|找人靠 QQ 群、表白墙|
|信息分散|竞赛、组队信息碎片化|错过报名、找不到队友|

### 1.2 项目愿景

**让每一个人的碎片化技能都能被看见、被激活、被变现，让每一个微小需求都能找到最合适的人。**

打造一个以校园为基础、以 0 佣金为核心、以算法匹配为引擎的轻量化技能互助平台，实现"人人皆可输出价值，微小需求皆有响应"。

### 1.3 核心价值主张

|价值点|说明|
|-|-|
|**0 佣金**|平台永久不抽成，用户私下交易，让劳动者获得 100% 收入|
|**强信任**|校园实名网络 + 信用评分体系 + 历史评价沉淀|
|**秒匹配**|多维度算法（标签+评分+校区+在线状态），3 秒推荐最优供给|
|**极轻量**|无需下载 APP，Web 端即开即用，发布技能只需 30 秒|
|**全免费**|平台对供需双方完全免费，不设付费墙|

### 1.4 产品定位

* **产品名称**：深技联（Shenzhen Technical Skills Union）
* **产品形态**：Web 响应式网站（PC 优先，适配移动端）
* **产品定位**：深圳大学校园垂直、永久 0 佣金、技能互助 + 算法智能匹配平台
* **目标用户**：深圳大学本科生、研究生、教职工、社团组织
* **第一阶段校区**：粤海校区、丽湖校区、沧海校区

### 2.2 核心用户画像

#### 画像 1：技能供给者（"斜杠青年"）

* **身份**：深大各院系学生，擅长 PPT、剪辑、翻译、设计、编程、笔记整理等
* **核心需求**：技能展示、低成本变现、积累口碑、竞赛组队
* **核心痛点**：无展示渠道、校外平台抽成高、陌生人信任差、匹配效率低
* **典型场景**：计算机系小王擅长 Python 脚本，想利用课余时间接单赚零花钱，但闲鱼/猪八戒抽成太高

#### 画像 2：需求发起者（"急单用户"）

* **身份**：有小额急单需求、竞赛备赛、组队找人的学生
* **核心需求**：快速找到靠谱的人、价格合理、校内优先
* **核心痛点**：QQ 群/表白墙信息杂乱、找不到合适的人、沟通成本高
* **典型场景**：经管系小李明天要交 PPT 作业但不会做，急需一个校内同学帮忙美化

#### 画像 3：竞赛参与者

* **身份**：参加挑战杯、互联网+、学科竞赛的学生
* **核心需求**：竞赛信息聚合、跨院系组队、资源模板、备赛指导
* **核心痛点**：信息分散在各学院官网、缺队友、缺资源

### 2.3 竞品分析

|维度|深技联|猪八戒/Upwork|闲鱼|QQ群/表白墙|
|-|-|-|-|-|
|佣金|**0%**|15-20%|0%|0%|
|信任机制|**信用分+实名**|平台担保|芝麻信用|无|
|匹配效率|**算法推荐**|搜索为主|搜索|随机|
|校园适配|**专属校区**|无|无|有但不结构化|
|竞赛服务|**一站式**|无|无|碎片信息|
|使用门槛|**极低**|高（需认证）|低|低|

**差异化优势**：深技联是唯一一个同时满足 **0 佣金 + 校园信任 + 算法匹配 + 竞赛服务** 的平台。

## 三、产品方案

### 3.1 核心功能模块

```
深技联平台
├── 用户系统       注册/登录、个人中心、信用评分
├── 技能广场       发布技能、浏览筛选、技能详情、达人推荐
├── 需求大厅       发布需求、算法匹配、需求详情
├── 订单系统       创建订单、状态流转、双方互评
├── 消息系统       私聊沟通、系统通知、轮询机制
└── 管理员后台     数据看板、用户管理、内容管理
```

#### 功能详情

**（1）技能广场**

* 用户发布个人技能（标题、标签、描述、价格、交付时间、校区）
* 支持按分类、校区、评分、价格区间筛选
* 热门技能推荐、优质达人榜
* 发布后即时展示，无需审核

**（2）需求大厅**

* 用户发布需求（标题、标签、描述、预算、截止时间、校区）
* 发布后自动触发算法匹配，推荐 3 个最优技能提供者
* 匹配算法权重：标签匹配 50% + 信用评分 20% + 校区 20% + 在线状态 10%

**（3）订单系统**

* 5 状态流转：待确认 → 进行中 → 待评价 → 已完成 / 已取消
* 双方确认后进入进行中，交付完成后互评
* 评价影响信用分，信用分影响匹配权重

**（4）消息系统**

* V1.0 采用轮询方式（每 5 秒拉取新消息）
* 支持聊天消息和系统通知
* 未读消息提醒

**（5）管理员后台**

* 数据看板（用户数、技能数、需求数、日活）
* 用户管理（查看、禁用、编辑）
* 内容管理（下架、删除）

### 3.2 核心业务流程

```
技能供给方                          需求发起方
    │                                  │
    ├─ 注册/登录 ──────────────────────┤
    ├─ 发布技能 ───────────────────────┤
    │    │                              │
    │    │         ┌──────────────┐     │
    │    └────────>│  技能广场    │<────┤
    │              │  需求大厅    │     │
    │              │  算法匹配    │     │
    │              └──────────────┘     │
    │                       │          │
    │                       ↓          │
    │              私聊沟通 → 确认交易   │
    │                       ↓          │
    │              线下/线上交付        │
    │                       ↓          │
    │              双方互评 → 信用分更新
    │                       ↓          │
    │              交易完成            │
```

### 3.3 信用评分体系

* 初始信用分：5.0 分
* 每次交易完成后双方互评（1-5 星）
* 信用分 = 所有评价的加权平均分
* 信用分 ≥ 4.8：进入"优质达人榜"，获得首页推荐
* 信用分 ≥ 3.0：可获得匹配算法权重
* 信用分 < 3.0：匹配算法中不计评分权重

### 4.4 匹配算法设计

```
总得分 = 标签匹配分 × 50% + 评分权重 × 20% + 校区权重 × 20% + 在线权重 × 10%
```

**标签匹配分**：需求标签与技能标签的交集占比 × 100  
**评分权重**：credit\_score ≥ 5.0 → 100 分；3.0-5.0 → 线性映射；< 3.0 → 0 分  
**校区权重**：同校区 → 100 分；不限 → 80 分；不同 → 50 分  
**在线权重**：在线 → 100 分；30 分钟内活跃 → 80 分；2 小时内 → 60 分；其他 → 0 分

匹配结果按总得分降序，返回前 3 个最优推荐。

### 4.5 安全设计

|安全措施|实现方式|
|-|-|
|密码加密|bcryptjs，成本因子 12|
|身份认证|JWT，Access Token 2 小时过期|
|接口限流|express-rate-limit，普通接口 60 次/分钟，登录 5 次/分钟|
|防重复提交|前端按钮置灰 + 后端唯一请求 ID|
|SQL 注入防护|Prisma 参数化查询|
|XSS 防护|输入过滤 + 输出转义|
|数据一致性|Prisma 事务保证原子操作|

## 八、风险评估与应对

|风险|可能性|影响|应对策略|
|-|-|-|-|
|**冷启动困难**（供需两端都不足）|高|高|种子用户策略；运营人员先发内容；降低发布门槛|

#### 改进 #3：公文通智能集成（V3.0）— 2026-06-05 ✅ 已部署

**动机**：纯技能交易平台存在冷启动问题，需要"即使没人发技能，也值得打开"的内容。公文通是深大学生每日必看的信息渠道，集成后能让网站成为信息入口。

**架构**：
```
Windows Task Scheduler (每天19:00)
  → run-sync.bat → npx tsx scripts/sync-announcements.ts
     → CAS 登录 + 爬取公文通前3页(约60条)
     → sourceId 去重 upsert
     → 写入 PostgreSQL (status='pending')

Claude Code Cron (每天19:05)
  → 读取 status='pending' 的公告
  → AI 分类: scholarship/lecture/exam/competition/activity/academic/recruitment/other
  → AI 提取: 活动时间、地点、截止日期、摘要
  → 筛选: 教师/行政/停电/天气 → status='discarded'
  → 更新 status='published'

网站前端 (消息页 Tab "📢 校园资讯")
  ← GET /api/announcements?category=&page=&pageSize=
  ← 分类筛选 + 加载更多
```

**数据流**：
- 首次运行爬取 60 条公告，AI 分类完成：33 条 published（8 类）+ 28 条 discarded
- published 分类分布：activity(11), academic(8), competition(4), lecture(3), recruitment(3), scholarship(2), exam(1), other(1)
- 30 天自动清理旧数据
